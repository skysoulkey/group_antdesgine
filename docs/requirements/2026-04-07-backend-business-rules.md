# 后端业务规则汇总 — 前端不可见逻辑

> 本文档整理前端 Mock 阶段已实现、但页面上不易直接看到的业务规则。
> 后端开发时必须落地这些逻辑，否则会出现权限越权、数据泄露、状态不一致等问题。

---

## 一、角色权限与数据隔离

### 1.1 三种角色的数据可见范围

| 角色 | 用户管理可见范围 | 通知管理可见范围 | 钱包可见范围 |
|------|----------------|----------------|-------------|
| 系统管理员 | 所有用户 | 所有通知 | 无钱包权限 |
| 集团管理员 | 本集团下的**公司管理员**（排除自己及其他集团管理员） | 本集团的通知 | 集团钱包 |
| 公司管理员 | **无权访问**用户管理 | 本公司的通知 | 公司钱包 |

### 1.2 集团管理员不可见/不可操作自己

- 用户管理列表中，后端返回数据时必须**过滤掉当前登录用户自身**及同级集团管理员
- 集团管理员只能查看、编辑、锁定/解锁**公司管理员**角色的用户
- 集团管理员不能修改自己的角色、状态、有效期、IP 限制等用户管理设定
- 基础信息页的 IP 限制为**只读展示**，不允许用户自行开关

### 1.3 集团管理员创建用户的限制

- 角色固定为「公司管理员」，不可选择其他角色
- 归属集团固定为当前登录集团（从 Token 解析），不可修改，不接受前端传入
- 仅可选择归属公司（当前集团下的公司列表）

### 1.4 系统管理员创建集团管理员的额外要求

- 必须绑定 APP 用户名
- APP 用户名格式校验：`^@[a-zA-Z0-9_]{2,30}$`（@ 开头，2-30 位字母/数字/下划线）
- APP 用户名全局唯一，不可与已有用户重复

### 1.5 编辑用户的角色限制

- 系统管理员：可修改用户角色
- 集团管理员：不可修改用户角色（字段只读）

### 1.6 公司管理员无用户管理权限

- 路由层面拦截 `/system/users`，返回 403
- 菜单层面不展示「用户管理」入口

## 二、用户状态机

### 2.1 多状态并存

一个用户可同时存在多种状态，互不冲突：

```
启用/停用（status）  ——  管理员手动设定
已过期（validPeriod） ——  系统自动判定，validPeriod < 当天 00:00:00
已锁定（isLocked）   ——  临时安全锁定，独立于启停用
```

- 只有 `status = 启用` 的用户才能执行锁定/解锁操作
- 已停用的用户不显示锁定/解锁按钮
- 账户有效期 `永久有效` 永不过期，自定义日期按天粒度判定

### 2.2 登录失败自动锁定

- 系统设置中 `loginFailMax`（默认 5 次）连续失败后自动锁定
- 锁定时长 `lockoutMinutes`（默认 30 分钟）
- 后端须记录连续失败次数，成功登录后清零

## 三、通知系统

### 3.1 通知渠道至少保留一种

对每个通知用户，站内/APP/邮件三个渠道不可同时关闭。后端保存偏好时须校验，全部为 false 则拒绝。

### 3.2 通知对象与渠道绑定

- APP/站内通知 → 通知对象为 APP 用户名（如 @miya_miya）
- 邮件通知 → 通知对象为邮箱地址（如 miya@cyberbot.sg）
- 后端发送通知时需根据渠道获取正确的目标地址

### 3.3 通知用户唯一性

- 同一通知配置中，APP 用户名不可重复
- 同一通知配置中，邮箱地址不可重复
- 新增和修改时都需校验

### 3.4 通知偏好同步

- 新增通知用户时，自动创建默认偏好（所有渠道开启）
- 删除通知用户时，级联删除其偏好记录
- 修改用户信息（姓名、APP 用户名、邮箱）时，偏好保持不变

### 3.5 站内通知状态

- 站内通知使用 `已读/未读` 状态（非 `已收到/异常`）
- 支持单条标记已读、全部标记已读
- 删除为软删除
- 默认保留 90 天（可在系统设置中配置），后端需定时清理任务

### 3.6 通知数据隔离

- 集团管理员：只能看到本集团的通知记录和通知配置
- 公司管理员：只能看到本公司的通知记录和通知配置
- 铃铛站内通知：每人只看自己的，未读数只统计自己的
- 所有查询接口从 Token 解析归属，禁止前端传入 group_id / company_id

## 四、钱包与资金操作

### 4.1 充值与转出的状态差异

- **充值**：创建后直接为「成功」状态，startTime 和 endTime 同时填充
- **转出**：创建后为「待审批」状态，endTime 留空，需审批通过后更新

### 4.2 MFA 强制校验

以下操作必须通过 MFA 验证（6 位数字）：
- 钱包充值
- 钱包转出
- 集团内部划转（下拨/调回）
- 修改绑定账号

前端 Mock 阶段任意 6 位通过，后端必须真实校验 TOTP。

### 4.3 内部划转余额计算

- 集团下拨：集团余额 - 金额，公司余额 + 金额
- 集团调回：集团余额 + 金额，公司余额 - 金额
- 后端须在同一事务中完成双方余额更新
- 单次划转只能选一种币种（USDT 或 PEA）

---

## 五、登录与安全

### 5.1 登录流程

```
用户名 + 密码 + 验证码 → 验证通过 → MFA 6位验证 → 登录成功
```

- 验证码为字母数字混合，不区分大小写
- MFA 为 TOTP（如 Google Authenticator），后端不可跳过

### 5.2 密码规则

- 最少 8 位
- 创建时需二次输入确认
- 修改密码时需验证旧密码

### 5.3 会话与密码过期

- 会话超时：1-72 小时（默认 8 小时），后端须强制踢出
- 密码过期：可配置天数（默认 90 天），0 为永不过期
- 过期后下次登录须强制修改密码

## 六、系统设置

### 6.1 安全设置（全局生效）

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| 登录失败次数上限 | 5 | 超过后自动锁定 |
| 锁定时长（分钟） | 30 | 自动解锁倒计时 |
| 会话超时（小时） | 8 | 无操作自动登出 |
| 密码过期（天） | 90 | 0=永不过期 |
| 强制 MFA | 关闭 | 开启后所有用户必须绑定 MFA |
| IP 白名单 | 关闭 | 系统级 IP 限制 |

### 6.2 通知渠道配置

- **站内通知**：启用开关 + 保留天数
- **邮件通知**：启用开关 + SMTP 服务器/端口/发件人（密码加密存储，API 返回脱敏）
- **Telegram**：启用开关 + Bot Token（加密存储）+ Chat ID
- **触发事件**：登录失败、大额转账、新企业注册、权限变更、系统错误、账户锁定

---

## 七、角色删除保护

- 如果某角色下还有关联账号（accountCount > 0），不允许删除
- 后端须检查关联数，前端已做提示但不可仅依赖前端校验

---

## 八、创建用户后的信息返回

- 创建成功后，后端须返回用户名和密码（或临时令牌）
- 前端弹窗展示：登录地址 + 用户名 + 密码，支持一键复制
- 此为一次性展示，密码不可再次查询（后端只存哈希）

---

## 九、数据隔离总原则

> 所有涉及集团/公司归属的查询接口，归属信息必须从登录 Token 中解析，**禁止前端传入 group_id / company_id**。前端不传这些参数，后端根据当前用户身份自动过滤。
