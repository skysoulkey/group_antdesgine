## 1. 概述

### 1.1 目标

制作一个独立的 HTML 长页面，可视化展示「持股企业追加投资/释放股份」从业务触发到审批闭环的完整流程。每一步配蓝图风格的页面 Mockup，关键区域高亮标注。

### 1.2 用途

- **给老板汇报**：全局视角，展示系统能力和信息流转
- **给公司管理员培训**：了解具体操作流程和场景

### 1.3 交付物

| 文件 | 说明 |
|------|------|
| `docs/flowcharts/approval-workflow.html` | 主文件，浏览器直接打开，不依赖项目运行环境 |
| `docs/flowcharts/assets/` | 步骤 1 的企业 APP 截图（用户提供） |

## 2. 页面结构

页面为竖向长页面，从上到下依次为：

### 2.1 页头

- 标题：「公司审批流程」
- 副标题：说明覆盖场景（追加投资 / 释放股份）
- 简洁商务风，与项目主色 #1677ff 一致

### 2.2 流程总览

精简版 6 步流程图，竖向排列，每步一个色块节点，步骤间用箭头连接。包含步骤 3 的左右分叉（命中规则 → 自动通过 / 未命中 → 人工审批）。作用是先给读者一个全局认知，再逐步展开细节。

### 2.3 步骤详情区域（6 个区块）

每个步骤区块的结构统一：

```
┌─────────────────────────────────────────┐
│  步骤编号 + 标题                          │
│  文字说明（1-2 句话描述这步发生了什么）      │
│                                         │
│  ┌─────────────────────────────────┐    │
│  │  页面 Mockup（蓝图风格）          │    │
│  │  关键区域用高亮框 + 标注箭头       │    │
│  └─────────────────────────────────┘    │
└─────────────────────────────────────────┘
```

## 3. 各步骤详细设计

### 步骤 1：业务触发 — 企业 APP 端发起操作

- **说明**：持股企业在自己的 APP 中发起追加投资或释放股份操作
- **Mockup 来源**：用户提供企业 APP 截图
- **呈现方式**：截图居中展示，下方文字标注「企业在自己的 APP 中发起操作，系统接收到事件后开始审批流程」
- **截图放置**：`docs/flowcharts/assets/` 目录下，页面通过相对路径引用

### 步骤 2：系统生成审批单 — 投资审批模块新增记录

- **说明**：系统自动在投资审批模块生成一条新的审批记录，状态为「待审批」
- **Mockup 内容**：投资审批列表页面
  - 顶部筛选栏：事件类型 Radio.Button（全部类型 / 持股企业追加投资 / 持股企业释放股份）+ 审批状态 Radio.Button（全部状态 / 待审批 / 已通过 / 已拒绝 / 自动通过 / 超时拒绝）
  - 表格列：订单时间、事件类型、企业名称、企业 ID、金额、货币单位、归属公司、归属公司 ID、审批状态、审批人昵称、审批时间、操作
  - 表格数据：4-5 行 Mock 数据，其中第一行为新增的「待审批」记录
  - **高亮标注**：第一行用高亮背景 + 边框标注，旁边加标注箭头「新增待审批记录」

### 步骤 3：自动审批规则匹配（分叉点）

- **说明**：系统检查该公司配置的自动审批规则，命中则自动通过，未命中则进入人工审批
- **Mockup 内容**：审批规则配置页面（审批规则 Tab）
  - 规则列表表格：创建时间、更新时间、规则名称、适用事件类型、适用范围、触发条件、启用状态、操作
  - 3 条 Mock 规则数据（如：「小额追加投资自动通过」金额 ≤ 10000、「CyberBot 企业白名单」指定企业、「组合规则」金额+企业）
  - **高亮标注**：标注规则的条件类型（金额 / 企业 / 组合）
- **分叉图**：Mockup 下方展示左右分叉
  - 左分支：命中规则 → 自动通过 → 发结果通知 → 流程结束
  - 右分支：未命中 → 待审批 → 进入步骤 4

### 步骤 4：三渠道通知分发

- **说明**：系统同时通过三个渠道通知公司管理员
- **呈现方式**：一个大框包裹三个并排的通知 Mockup
- **Mockup 内容**：
  - **小程序通知**：模拟聊天消息界面，消息气泡 + 底部 Inline Keyboard 按钮（✅ 同意 / ❌ 拒绝）。参考 Telegram Bot 消息样式
  - **邮件通知**：模拟邮件界面，HTML 格式，包含标题、正文、跳转链接按钮。使用需求文档中定义的邮件模版文案
  - **站内通知**：模拟通知列表界面，一条未读通知高亮显示，可点击跳转
- **高亮标注**：小程序通知的 Inline Keyboard 按钮区域重点标注「可直接在小程序内审批」

### 步骤 5：公司管理员审批操作（三条路径）

- **呈现方式**：三条路径并排展示

#### 路径 A — 小程序直接审批

- Mockup 1：操作前 — 消息带两个可点击按钮
- Mockup 2：操作后 — 按钮变灰，文字变为「已同意 ✓」或「已拒绝 ✗」
- Mockup 3：聊天室收到审批结果通知新消息（即步骤 6 的内容在聊天室中的展现）
- **高亮标注**：按钮状态变化 + 结果通知消息

#### 路径 B — Web 端审批

- Mockup 1：审批列表页面，鼠标悬停在「通过」按钮上
- Mockup 2：二次确认弹窗（「确认通过 XXX 的持股企业追加投资申请？」，含可选备注输入框）
- **高亮标注**：操作按钮 + 确认弹窗

#### 路径 C — 超时未处理

- Mockup：小程序消息中按钮已移除，消息追加「⏰ 已超时」
- 文字说明：截止时间内无人操作 → 系统自动拒绝

### 步骤 6：审批结果通知

- **说明**：不论通过哪条路径完成审批，系统自动向所有通知对象发送结果通知（三渠道同步）
- **呈现方式**：三种结果并排展示
  - ✅ 已通过：通知内容包含企业名称、事件类型、操作人、操作时间
  - ❌ 已拒绝：通知内容包含企业名称、事件类型、操作人、操作时间
  - ⏰ 超时拒绝：通知内容包含企业名称、事件类型、系统自动标记
- **补充说明**：路径 A 的结果通知直接以新消息形式出现在聊天室中

## 4. 视觉规范

### 4.1 整体风格

- **蓝图风格**：有色彩和布局，关键数据用 Mock 数据填充，不追求像素级还原
- 主色：Ant Design 蓝 `#1677ff`
- 重要区域用高亮框（橙色 `#fa8c16` 边框）+ 标注箭头引导视线
- 商务、大气、简洁，与项目整体设计风格一致

### 4.2 Mockup 规范

- 基于 Ant Design 组件风格（Table、Radio.Button、Button、Modal、Tag、Card）
- 表格使用斑马纹（奇偶行交替背景色）
- 筛选栏使用 Radio.Button 填充风格（蓝底白字选中态）
- 弹窗使用圆角卡片 + 阴影
- 小程序消息使用聊天气泡风格

### 4.3 高亮标注规范

- 高亮框：2px 实线橙色边框 `#fa8c16`，圆角 4px
- 标注箭头：橙色箭头 + 文字说明
- 新增记录行：浅橙色背景 `#fff7e6`

### 4.4 步骤间连接

- 步骤之间用蓝色箭头连接
- 分叉点用左右分支展示，分支线用不同颜色区分（绿色=命中、蓝色=未命中）
- 汇聚点明确标注回到主流程

## 5. 技术实现

### 5.1 技术方案

- 纯 HTML + 内联 CSS，单文件自包含
- 不依赖任何外部 CDN、框架或项目运行环境
- 图片资源通过相对路径引用（`assets/` 目录下）
- 所有 Mockup 用 HTML/CSS 直接绘制，不使用图片

### 5.2 文件结构

```
docs/flowcharts/
├── approval-workflow.html    # 主页面
└── assets/
    └── step1-app-screenshot.png  # 用户提供的企业 APP 截图
```

### 5.3 兼容性

- 目标：现代浏览器（Chrome、Safari、Edge）
- 支持打印（Print 友好的样式）
- 响应式：在 1200px+ 屏幕上最佳展示
